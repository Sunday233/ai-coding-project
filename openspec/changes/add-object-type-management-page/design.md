## Context

当前需求基于对象类型管理界面的视觉稿：左侧导航和顶部面包屑是公共组件，中间区域是随路由切换的业务页面。项目约束为 Vue 3 + TypeScript + Vite + Ant Design Vue + Scoped Scss + Composition API，并采用通用 Vue3 目录结构（包含 services/types 分层）。

当前尚未进入实现阶段，首要目标是形成可实施的页面能力定义与技术方案，确保开发、联调与验收使用同一标准。

## Goals / Non-Goals

**Goals:**
- 建立可复用的管理端壳层布局，统一侧边导航、顶部面包屑和内容区域承载方式。
- 实现对象类型管理列表页能力定义：搜索、列表展示、状态与可见性标记、分页、查看动作。
- 统一列表页交互约束（加载态、空态、错误态、分页切换与选择行为），提升后续页面扩展一致性。
- 形成可直接转化为开发任务和测试用例的规格文档。

**Non-Goals:**
- 不在本变更中覆盖对象类型的新增、编辑、删除完整流程。
- 不在本变更中重做整体视觉设计或替换 UI 组件库。
- 不在本变更中定义后端实现细节（仅定义前端可依赖的数据契约与交互行为）。

## Decisions

### Decision 1: 使用统一壳层布局承载管理页面
- 方案：将左侧导航与顶部面包屑抽离为公共布局组件，内容区通过路由视图切换业务页。
- 理由：减少页面重复结构，统一导航行为与样式，降低后续新增管理页成本。
- 备选方案：每个页面自行实现导航和面包屑。
- 未选原因：重复代码多，样式与交互易漂移，维护成本高。

### Decision 2: 列表页按“页面容器 + 表格展示”分层
- 方案：页面容器负责查询条件与分页状态，表格展示组件负责列渲染与交互事件透传。
- 理由：便于复用列表展示逻辑，并让查询状态管理更清晰。
- 备选方案：单文件页面承载全部逻辑与渲染。
- 未选原因：当字段和交互增长时可读性变差，测试粒度过大。

### Decision 3: 数据访问统一走 services + types
- 方案：接口调用统一在 `src/services`，请求/响应与领域模型在 `src/types`。
- 理由：符合项目规范，避免页面层直接处理原始接口结构，便于后续联调和类型演进。
- 备选方案：页面内直接 mock 或直接发请求。
- 未选原因：耦合严重，不利于复用和测试。

### Decision 4: 列表关键状态采用规范化映射
- 方案：状态（如“活跃/建设中”）与可见性（如“可见/不可见”）通过枚举或常量映射统一渲染。
- 理由：防止文案分散和颜色不一致，便于国际化及主题适配。
- 备选方案：在模板中硬编码标签文案和颜色。
- 未选原因：易产生不一致，维护成本高。

## Risks / Trade-offs

- [风险] 设计稿字段与后端真实字段不一致，导致联调阶段返工。
  -> Mitigation：在规格中先定义字段契约与可选字段，联调前进行字段映射评审。
- [风险] 公共壳层组件职责过重，后续扩展时出现耦合。
  -> Mitigation：明确壳层只负责结构与导航，不承载业务状态。
- [风险] 列表交互规则不明确（如勾选跨页是否保留），影响实现一致性。
  -> Mitigation：在 specs 中以 SHALL 语句固化行为并配套场景。
- [权衡] 先按规范抽象分层会增加初期文件数量。
  -> Mitigation：通过统一目录模板和任务分解降低落地成本，换取长期维护收益。

## Migration Plan

1. 建立管理端壳层布局与路由承载结构，保证中间页面可切换。
2. 定义对象类型列表页的数据模型、查询参数与服务层接口。
3. 实现对象类型列表页（搜索、表格、标签、分页、查看动作）及状态处理。
4. 按规格场景执行页面验收，补齐边界态（空态/错误态）。
5. 预留与真实 API 联调替换点，确保 mock 到真实接口迁移平滑。

回滚策略：若上线验证失败，可先回退对象类型页面路由入口，保留公共壳层不下线；必要时回退整个变更分支。

## Open Questions

- “查看”动作对应的是详情页路由还是弹窗侧栏？
- 分页每页条数候选值是否固定为 10/20/50，是否需要记忆用户偏好？
- 搜索是否仅按“类型名称”模糊匹配，是否需要支持状态/日期筛选？
- 可见性字段是否允许在列表页直接切换，还是仅展示只读标签？
